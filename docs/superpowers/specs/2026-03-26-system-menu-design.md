# System Menu 설계 문서

**날짜:** 2026-03-26
**모듈:** system
**범위:** 어드민 메뉴 조회 API 및 DB 설계

---

## 개요

어드민 사이드바에 표시할 메뉴를 DB로 관리하고, 트리 구조로 조회하는 API를 제공한다.
현재 2단계(대메뉴 > 소메뉴) 구조이며, 추후 3단계 이상으로 확장 가능하도록 self-referencing 단일 테이블로 설계한다.

---

## DB 설계

### 테이블: `ST_MENU_BASE`

```sql
CREATE TABLE ST_MENU_BASE (
    MENU_NO         BIGINT          GENERATED BY DEFAULT AS IDENTITY,
    PARENT_MENU_NO  BIGINT          NULL,           -- 상위메뉴번호 (NULL = 대메뉴)
    MENU_NM         VARCHAR(200)    NOT NULL,        -- 메뉴명
    MENU_URL        VARCHAR(2000)   NULL,            -- 연결 URL (대메뉴는 NULL 가능)
    SORT_ORD        INTEGER         NOT NULL DEFAULT 0,
    USE_YN          CHAR(1)         NOT NULL DEFAULT 'Y',
    REG_DTM         TIMESTAMP       NOT NULL DEFAULT NOW(),
    REG_ID          VARCHAR(50)     NOT NULL,
    MOD_DTM         TIMESTAMP       NOT NULL DEFAULT NOW(),
    MOD_ID          VARCHAR(50)     NOT NULL,
    CONSTRAINT PK_ST_MENU_BASE PRIMARY KEY (MENU_NO),
    CONSTRAINT FK_ST_MENU_BASE_SELF FOREIGN KEY (PARENT_MENU_NO)
        REFERENCES ST_MENU_BASE (MENU_NO)
);
```

**설계 결정:**
- `MENU_LVL` 컬럼 미포함 — 대메뉴 판별은 `PARENT_MENU_NO IS NULL`, 깊이는 재귀 CTE에서 동적 계산
- `GENERATED BY DEFAULT AS IDENTITY` 사용 — 초기 데이터 INSERT 시 명시적 ID 지정 가능
- 역할(ROLE) 기반 메뉴 접근 제어는 추후 별도 설계

## API 설계

### 엔드포인트

```
GET /api/admin/system/menus
```

- 인증 필요 (JWT)
- `USE_YN = 'Y'` 메뉴만 반환
- `MenuController`는 `@RequestMapping("/api/admin/system/menus")` 선언 (`AdminAuthController`가 `/api/admin/system`을 사용 중이므로, 충돌 방지를 위해 전체 경로를 클래스 레벨에 선언)
- `OPS` 역할의 접근 허용 여부는 추후 역할 기반 제어 설계 시 결정 (현재는 `SecurityConfig`의 `/api/admin/**` 허용 역할 기준 따름)

### 응답

## 백엔드 구현 설계

### 파일 목록

| 파일 | 위치 |
|------|------|
| `Menu.java` | `model/system/` |
| `MenuMapper.java` | `mapper/system/` |
| `MenuMapper.xml` | `resources/mapper/system/` |
| `MenuResponse.java` | `dto/system/` |
| `MenuService.java` | `service/system/` |
| `MenuController.java` | `controller/system/` |

### SQL 전략

`MenuMapper.xml`에서 `USE_YN = 'Y'` 전체를 flat하게 조회 후, `MenuService`에서 트리로 조립한다.
재귀 CTE를 사용하지 않고 단순 전체 조회로 구현 — 메뉴 수가 적고, 트리 조립은 Java에서 처리.

```sql
SELECT MENU_NO, PARENT_MENU_NO, MENU_NM, MENU_URL, SORT_ORD
FROM ST_MENU_BASE
WHERE USE_YN = 'Y'
ORDER BY SORT_ORD
```

> 추후 3단계+ 및 역할 기반 필터링이 추가될 경우 재귀 CTE로 전환 검토.

### DTO 설계 (`MenuResponse`)

`children` 리스트를 Service에서 동적으로 추가해야 하므로 `record` 대신 `@Getter + @Builder + @Setter(children only)` 또는 생성자에서 mutable list를 허용하는 방식으로 구현.

```java
@Getter
@Builder
public class MenuResponse {
    private Long menuNo;
    private String menuNm;
    private String menuUrl;
    private int sortOrd;
    @Builder.Default
    private List<MenuResponse> children = new ArrayList<>();
}
```

### 트리 조립 로직 (Service)

1. Mapper에서 flat 리스트 전체 조회
2. `Map<Long, MenuResponse>`로 인덱싱 (key = MENU_NO)
3. `PARENT_MENU_NO`가 null인 항목 → 루트 목록 구성
4. 나머지 항목 → 부모 MenuResponse의 `children`에 추가
5. 각 `children` 리스트를 `SORT_ORD` 기준으로 정렬 (자식 추가 완료 후)
6. 루트 목록도 `SORT_ORD` 기준 정렬 후 반환

## 추후 고려 사항

- 역할(ROLE) 기반 메뉴 접근 제어: 메뉴-역할 매핑 테이블(`ST_MENU_ROLE`) 추가 예정
- `OPS` 역할 접근 허용 범위 결정
- 메뉴 관리 CRUD API (추가/수정/삭제/순서변경)
- 3단계 이상 메뉴 추가 시 재귀 CTE 전환 검토
